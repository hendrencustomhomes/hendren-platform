### `/more/catalog/[id]` → `src/app/more/catalog/[id]/page.tsx`

**Bug found and fixed.** The route segment is named `[id]` and receives `catalog_sku` as the dynamic value. This is acceptable — renaming to `[catalogSku]` would be a cosmetic improvement only and is deferred.

### Source count

Source count is computed client-side by fetching all `catalog_sku` values from `pricing_rows`:

```ts
supabase.from('pricing_rows').select('catalog_sku').not('catalog_sku', 'is', null)
```

Null safety: ✅ `.not('catalog_sku', 'is', null)` excludes nulls at query level; the client-side loop additionally guards with `if (!sku) continue`.

**Scaling note (not a bug):** This fetches all linked pricing row SKUs on every catalog page load. At current scale this is acceptable. At high volume (thousands of pricing rows), this should be replaced by a DB-side aggregate (RPC or view returning `count` per `catalog_sku`). Document for future cleanup.

### Access gating

Access check at init: ✅ `getCurrentPricingAccess('catalog')` called on mount.

**Bug found and fixed.** The `s.error` state was set correctly when access was denied or data loading failed, but `CatalogPageClient` never rendered it. The component showed an empty list with no message. Users with denied access would see a blank page.

**Fix applied:** Added `{s.error && <div ...>{s.error}</div>}` above the loading/list area.

### Loading state leak

**Bug found and fixed.** `load()` called `setLoading(true)` at the start but only called `setLoading(false)` on the success path. If `load()` threw after `setLoading(true)`, the caller's catch block would set the error but `setLoading(false)` would never be called in the context of `handleCreate`. The page would be stuck in loading state with an error message.

The `init()` caller correctly called `setLoading(false)` in its own catch. But `handleCreate` called `load()` with no such safety net for the loading state.

**Fix applied:** Wrapped `load()` body in `try/finally` so `setLoading(false)` is always called regardless of success or failure. `init()`'s catch continues to call `setLoading(false)` as well — harmless double-call on failure, fine.

## Real Bugs Found

### Bug 1 — Params not async/awaited in `[id]/page.tsx`

**File:** `src/app/more/catalog/[id]/page.tsx`

Next.js 16 requires dynamic route params to be typed as `Promise<{...}>` and awaited in server components. The catalog detail route was not async and accessed `params.id` synchronously. `catalogSku` would receive the Promise object string representation at runtime rather than the actual SKU segment.

**Fix:** Route made `async`, params typed as `Promise<{ id: string }>`, destructured with `await params`. Matches the existing pattern in `price-sheets/[id]/page.tsx` and `bids/[bidId]/page.tsx`.

---

### Bug 2 — `CatalogPageClient` silently swallows error state

**File:** `src/app/more/catalog/_components/CatalogPageClient.tsx`

`useCatalogPage` correctly sets `s.error` on load failure and on access denial. `CatalogPageClient` never rendered it. Access-denied users and users hitting load errors would see an empty catalog list with no message.

**Fix:** Added error display above the loading/list area.

---

### Bug 3 — `load()` leaks loading state on throw inside `handleCreate`

**File:** `src/app/more/catalog/_hooks/useCatalogPage.ts`

`load()` set `setLoading(true)` at entry and `setLoading(false)` only on success. When called from `handleCreate` after a successful create, a subsequent `load()` failure would leave the page permanently stuck showing the loading state. `handleCreate`'s catch handled the error message but had no path to call `setLoading(false)`.

**Fix:** Wrapped `load()` body in `try/finally { setLoading(false) }` so loading is always cleared regardless of outcome.

---

### Bug 4 — `CatalogDetailPage` missing access gate

**File:** `src/app/more/catalog/_components/CatalogDetailPage.tsx`

Any user who navigated directly to `/more/catalog/[sku]` could view the catalog identity and all linked source pricing rows without any permission check. The list page gated on `getCurrentPricingAccess('catalog')` but the detail page performed no access check.

**Fix:** Added `getCurrentPricingAccess('catalog')` as the first `await` inside the load `try` block. Returns early with error state if `canView` is false. The existing `finally` ensures loading clears on early return.

---

### Bug 5 — `item.title` absent from identity section

**File:** `src/app/more/catalog/_components/CatalogDetailPage.tsx`

`title` is the primary identity label for a catalog item. It appeared only in the Nav `<title>` attribute. The identity card body showed SKU and description but not title as an explicit labeled field.

**Fix:** Added `<div><strong>Title:</strong> {item.title}</div>` between SKU and Description in the identity section.
