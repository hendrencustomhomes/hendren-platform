## 5. Integration Integrity — One Issue Found (Orphaned Load)

### Pricing rows reference catalog correctly

`rows.ts` calls `getCatalogItemBySku` when `catalog_sku` is provided. It uses the result to:
- Fall back on `catalogItem.title` for `description_snapshot`
- Use `catalogItem.cost_code_id` for the row's cost code
- Use `catalogItem.default_unit` as a fallback for `unit`
- Store `catalogItem.catalog_sku` on the row

This is correct one-directional integration.

### `catalogItems` is loaded but never consumed (dead fetch)

`usePricingWorksheetPersistence` fetches the catalog item list on every pricing worksheet page load and returns it from the hook, but `PricingWorksheetPageOrchestrator` never destructures or passes it anywhere.

```ts
// Persistence hook — fetches but result goes nowhere
const catalogRows = await listCatalogItems(supabase, { cost_code_id: headerRow.cost_code_id, is_active: true })
setCatalogItems(catalogRows)
// ...
return { ..., catalogItems, ... }

// Orchestrator — never accesses persistence.catalogItems
```

This was likely scaffolded for a catalog-search / row-linking picker UI that was never built. It adds a DB query to every pricing worksheet page load with no current benefit.

**Not fixed in this pass** — modifying the pricing module for this cleanup is out of scope here. Document for the next pricing cleanup pass.

## 6. Bugs Found and Fixed (Pricing Module — Surfaced by Integration Audit)

Both bugs were in `src/lib/pricing/revisions.ts` in the row-copy loop that runs during `createPricingHeaderRevision`.

### Bug A — `pricing_type` lost on revision create

`createPricingRow` was called without `pricing_type`. All rows in the new revision would default to `'unit'` regardless of their original type. Any `lump_sum` or `allowance` rows would silently revert.

### Bug B — `quantity` lost on revision create

`createPricingRow` was called without `quantity`. All rows in the new revision would get `quantity: null` regardless of original values.

**Fix applied:** Both `pricing_type: row.pricing_type` and `quantity: row.quantity` added to the `createPricingRow` call in the revision row-copy loop.

```diff
 const newRow = await createPricingRow(supabase, {
   pricing_header_id: newHeader.id,
   catalog_sku: row.catalog_sku,
   vendor_sku: row.vendor_sku,
   description_snapshot: row.description_snapshot,
+  pricing_type: row.pricing_type,
+  quantity: row.quantity,
   unit: row.unit,
   unit_price: row.unit_price,
   lead_days: row.lead_days,
   notes: row.notes,
   is_active: row.is_active,
 })
```

These follow the same pattern as the `pricing_type` omission fixed in the previous session (`usePricingWorksheetPersistence.ts`).

## 7. Risks

### Real

- **Revision row copy was losing `pricing_type` and `quantity`** — fixed in this pass. Any revision created before this fix would have incorrect row types and missing quantities. Historical data in the DB is not retroactively corrected by this fix.

- **Catalog has no access-gated UI** — the permission row exists but no route enforces it. Until the catalog UI is built, the permission row is inert.

### Noted (not urgent)

- `catalogItems` dead load adds an unnecessary DB query on every pricing worksheet open. Low performance impact at current scale, but should be cleaned up when the catalog-linked row picker is built.

## 8. Required Structure for Catalog UI (Do Not Build Yet)

When Catalog UI is implemented, it should follow the module standard exactly.

```
src/app/more/catalog/
  page.tsx                          — thin: delegates to CatalogPageClient
  [id]/
    page.tsx                        — thin: delegates to CatalogDetailPage
  _components/
    CatalogPageClient.tsx           — list + inline create, access-gated
    CatalogDetailPage.tsx           — view + edit fields
  _hooks/
    useCatalogPage.ts               — list state, filter state, create flow
    useCatalogDetail.ts             — load single item, edit state, save
```

The pattern should mirror `PricingHeadersPageClient` / `usePricingHeadersPage` in shape — a thin page entry, a client component, and a local hook owning all state and DB calls.

Domain operations are already available in `src/lib/pricing/catalog.ts`. No new domain work required.

`src/app/more/page.tsx` needs a Catalog entry in `MORE_ITEMS`.

Access gating should use `getCurrentPricingAccess('catalog')` or equivalent from `@/app/actions/pricing-access-actions`.
